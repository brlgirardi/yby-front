Na classificação Atomic Design (Brad Frost, 2016, atomicdesign.bradfrost.com), `Tag` é um **átomo** — elemento indivisível usado por moléculas e organismos. Ver `Foundations/Atomic Design` para o restante da classificação. Este documento descreve a regra de negócio que esse átomo carrega: o vocabulário de status do domínio sub-adquirência e como cada nome se mapeia para tom visual, ícone e ação esperada do usuário.

A interface usa **um único átomo de status semântico**, o `Tag`, parametrizado pelo nome do status. O componente resolve internamente:

- **Tom semântico** (sucesso, processamento, futuro, aviso, erro, neutro)
- **Cor de fundo, texto e borda** (paleta AntD adotada)
- **Ícone correspondente** (check, loader, clock, alert, x, minus, info)

Essa decisão é deliberada: o backend e os mocks descrevem status em **português, livre de cor**. A regra de cor vive no front, no `STATUS_MAP` de `Tag.tsx`. Isso evita acoplar a semântica visual à API.

## Sistema semântico — 6 tons

| Tom | Quando usar | Paleta (bg / texto / borda) | Ícone padrão |
|---|---|---|---|
| **success** | Operação concluída, nada a fazer | `#F6FFED` / `#237804` / `#B7EB8F` | check |
| **processing** | Sistema executando, usuário aguarda | `#E6F7FF` / `#003A8C` / `#91D5FF` | loader |
| **future** | Evento previsto ou agendado, sem ação | `#F9F0FF` / `#531DAB` / `#D3ADF7` | clock |
| **warning** | Requer atenção ou decisão do usuário | `#FFFBE6` / `#874D00` / `#FFE58F` | alert |
| **error** | Falha ou bloqueio definitivo | `#FFF1F0` / `#820014` / `#FFCCC7` | x |
| **neutral** | Operação encerrada sem efeito | `#F5F5F5` / `rgba(0,0,0,0.55)` / `#D9D9D9` | minus |

> Tons baseados nas paletas AntD (Polar Green, DayBreak Blue, Geek Purple, Calendula Gold, Dust Red, Neutral). A escolha foi feita para alinhar com o `globals.css` (variantes `.kpi-*`) e com a paleta Tupi.

## Inconsistências detectadas

Pontos a normalizar — não bloqueiam uso, mas pedem atenção do time:

- **Duplicação masculino/feminino:** `Aprovado`/`Aprovada`, `Recusado`/`Recusada`, `Cancelado`/`Cancelada`, `Estorno`/`Estornada`. O `STATUS_MAP` resolve via aliases, mas a API/mocks deveriam padronizar.
- **Conciliação com 3 formas para "parcial":** `Parc. Conciliado`, `Parcialmente conciliado`, `Parcialmente liquidado`. Origem é o vocabulário Tupi de conciliação. Recomendado escolher uma forma canônica por contexto.
- **Mapeamento ad-hoc em features:** vários componentes traduzem `row.status` (string vinda do mock) para um status Tupi (`Sucesso → Pago`, `Falha → Erro`, `Concluído → Liquidado`, `Crédito Vendido → Antecipado`). Esse mapeamento está espalhado (`EcCobrancas.tsx`, `EcLinksPagamento.tsx`, `EcLiquidacoes.tsx`). Centralizar num utilitário (ex.: `src/lib/status/translate.ts`) reduz divergência.
- **"Em processamento" vs "Em análise":** semanticamente próximos. "Em análise" indica revisão manual; "Em processamento" indica execução automática. Confirmar uso antes de unificar.
- **`Tag` da AntD vs `Tag` shared:** `SettlementsTab`, `StatementTab`, `AgendaFunding`, `AgendaInstallments` importam o `Tag` da AntD (label sem semântica) e o `StatusTag` (alias para o `Tag` shared) lado a lado. A migração completa para o `Tag` shared está pendente.

## Como adicionar um status novo

1. Defina o **tom semântico** apropriado (success / processing / future / warning / error / neutral).
2. Adicione a entrada em `STATUS_MAP` em `src/components/shared/Tag.tsx`, mapeando para o tom e ícone.
3. Adicione o nome ao tipo `TagVariant` no topo do arquivo.
4. Atualize esta página com nome, significado, telas onde aparece e preview.
5. Atualize a story `Design System/Atoms/Tag` adicionando o status na galeria.
