# Atomic Design

A metodologia Atomic Design (Brad Frost, 2016) organiza a interface em **cinco camadas hierárquicas**, do menor elemento indivisível ao layout completo de uma página. O Yby Front adota essa estrutura como linguagem comum entre design e engenharia: quando o time discute "Tag", está no nível dos átomos; quando discute "DataTable", está no nível dos organismos. Este documento descreve cada camada, os critérios de classificação adotados pelo projeto, e mapeia os componentes existentes nas pastas `src/components/`.

> "Atoms, molecules, organisms, templates, and pages concurrently working together to create effective interface design systems." — Brad Frost, *Atomic Design* (atomicdesign.bradfrost.com, 2016)

## Por que adotar

- **Vocabulário compartilhado** — design e engenharia falam a mesma língua ao discutir um componente.
- **Reuso explícito** — átomos são compostos em moléculas; moléculas em organismos. A árvore deixa óbvio onde reutilizar.
- **Storybook como verdade única** — cada átomo, molécula e organismo tem sua story; templates e páginas referenciam essas stories.
- **Refatoração guiada** — quando um átomo muda (ex.: `Tag`), o impacto sobe a cadeia (moléculas → organismos) de forma previsível.

## Critérios de classificação

A classificação no Yby segue regras objetivas, não interpretação subjetiva:

| Camada | Regra de inclusão | Sinais práticos |
|---|---|---|
| **Átomo** | Elemento de UI indivisível. Não compõe outros componentes do projeto. | Renderiza HTML/SVG primitivo ou wrapper fino de AntD. Sem estado complexo. Sem fetch. |
| **Molécula** | Combina 2+ átomos para criar uma unidade funcional pequena. | Costuma ter um label + input, ou um título + ação, ou cabeçalho + corpo. |
| **Organismo** | Composição complexa de moléculas e átomos que entrega uma fatia funcional. | Tabela com filtros + paginação + ações; formulário inteiro; sidebar inteira. |
| **Template** | Layout estrutural reutilizável que define onde organismos aparecem. | Define grid/regiões, mas o conteúdo é placeholder. |
| **Página** | Instância concreta de um template, com dados reais (mocks por enquanto). | Vive em `src/app/` (App Router) ou `src/features/`. |

Regras adicionais adotadas pelo projeto:

- Componentes que envolvem apenas estilização visual de um AntD primitivo continuam como **átomos** (ex.: `Button`, `Tooltip`).
- Componentes de layout estrutural (`AppShell`, `ScreenRouter`) são **templates**, não organismos, mesmo compondo muitas peças — sua função é orquestrar regiões, não entregar conteúdo.
- Rotas em `src/app/(app)/` e features em `src/features/{persona}/{version}/` são **páginas**.

## Camada 4 — Templates

Layouts estruturais que orquestram regiões e onde os organismos aparecem. Não carregam dados de domínio.

| Componente | Caminho | Função |
|---|---|---|
| AppShell | `src/components/layout/AppShell.tsx` | Header + Sidebar + área de conteúdo. Template raiz da área autenticada. |
| ScreenRouter | `src/components/layout/ScreenRouter.tsx` | Decide qual feature renderiza dentro do shell para a persona/versão ativa. |
| SidebarRouter | `src/components/layout/SidebarRouter.tsx` | Decide qual sidebar (`EcSidebar`/`AqSidebar`/`SubV1Sidebar`) renderiza por persona. |
| AuthGuard | `src/components/layout/AuthGuard.tsx` | Template de proteção — redireciona se não autenticado. |

## Como adicionar um novo componente

1. **Decida a camada** — aplique a regra de inclusão da tabela acima.
2. **Crie o arquivo no domínio adequado** — hoje a estrutura é por domínio (`shared`, `ui`, `layout`, `chargeback`, etc.); a Fase 2 do plano move para `atoms/molecules/organisms/templates`.
3. **Crie a story junto** — toda peça nova entra no Storybook como verdade única.
4. **Documente status novos aqui** — se o componente introduz um status que ainda não existe em `Tag`, atualize também `Foundations/Regras de Negócio/Status & Tags`.
